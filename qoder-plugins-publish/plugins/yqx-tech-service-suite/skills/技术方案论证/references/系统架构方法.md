# 系统架构方法

## 一、架构分解原则（MECE）

### 1.1 MECE原则定义
MECE（Mutually Exclusive, Collectively Exhaustive）即"相互独立，完全穷尽"。在系统架构分解中应用此原则，确保每个模块边界清晰、无重叠、无遗漏。

### 1.2 分解层次

```
L0: 产品整机（系统级）
├── L1: 子系统（按功能域划分）
│   ├── L2: 模块（按职责划分）
│   │   ├── L3: 组件（按实现单元划分）
│   │   └── L3: 组件
│   └── L2: 模块
└── L1: 子系统
```

### 1.3 分解维度选择

| 维度 | 适用场景 | 示例 |
|------|----------|------|
| 功能域 | 按业务功能划分 | 电源管理、信号处理、通信模块 |
| 物理域 | 按物理结构划分 | 壳体、PCBA、电池包、显示屏 |
| 数据流 | 按数据流向划分 | 采集层、处理层、传输层、应用层 |
| 控制流 | 按控制逻辑划分 | 感知、决策、执行、反馈 |

### 1.4 模块定义规范

每个模块需明确以下要素：

```markdown
### 模块：[模块名称]

| 项目 | 内容 |
|------|------|
| 模块ID | MOD-XXX |
| 所属子系统 | |
| 功能边界 | 做什么：[核心职责]；不做什么：[明确排除] |
| 输入接口 | [信号/数据/能量的输入规格] |
| 输出接口 | [信号/数据/能量的输出规格] |
| 性能要求 | [响应时间、精度、容量等] |
| 技术约束 | [尺寸、重量、功耗、成本上限等] |
| 依赖模块 | [上游/下游依赖] |
| 关键器件 | [核心器件清单] |
```

### 1.5 分解质量检查
- [ ] 同一层级所有模块是否覆盖父级全部功能？
- [ ] 同级模块之间是否存在功能重叠？
- [ ] 每个模块是否可独立设计、测试、替换？
- [ ] 模块数量是否在7±2的认知负荷范围内？
- [ ] 分解深度是否一致（同级模块层级对齐）？

## 二、接口定义方法

### 2.1 接口分类

| 接口类型 | 描述 | 定义要素 |
|----------|------|----------|
| 硬件接口 | 物理连接器、电气信号 | 引脚定义、电气规格、机械尺寸 |
| 软件接口 | API、协议、数据格式 | 输入参数、返回值、错误码、时序 |
| 机械接口 | 装配关系、公差配合 | 配合尺寸、定位基准、紧固方式 |
| 人机接口 | 用户交互界面 | 交互方式、信息架构、反馈规则 |

### 2.2 接口定义文档（IDD）模板

```markdown
### 接口：[接口名称]

| 项目 | 内容 |
|------|------|
| 接口ID | INT-XXX |
| 接口类型 | 硬件/软件/机械/人机 |
| 提供方 | [模块A] |
| 消费方 | [模块B] |
| 通信方向 | 单向/双向/广播 |

#### 规格定义
- [具体规格参数]

#### 时序要求
- [触发条件、响应时间、超时处理]

#### 错误处理
- [异常场景与容错机制]

#### 版本管理
- [兼容性策略与变更历史]
```

### 2.3 接口设计原则
1. **最小化原则**：接口暴露最少必要信息
2. **稳定性原则**：接口一旦发布尽量保持稳定
3. **可测试原则**：每个接口可独立验证
4. **文档化原则**：接口变更必须同步更新文档

## 三、技术路线对比维度

### 3.1 对比矩阵维度

| 维度 | 说明 | 评估方法 |
|------|------|----------|
| 技术原理 | 核心工作机制 | 文献调研、专利分析 |
| 性能上限 | 理论/实测可达到的性能指标 | 对标测试、 datasheet |
| 技术成熟度 | TRL等级（1-9级） | 行业应用案例统计 |
| BOM成本 | 物料清单成本 | 询价、公开报价 |
| 供应链可得性 | 供应商数量、交货周期、地域分布 | 供应商调研 |
| 专利壁垒 | 核心专利覆盖情况、授权成本 | 专利检索分析 |
| 扩展性 | 向相关产品/性能升级的兼容能力 | 技术演进路线评估 |
| 人才可得性 | 团队是否具备相关能力或是否可招聘 | 招聘市场调研 |
| 生态支持 | 开发工具、社区、第三方资源 | 生态活跃度评估 |

### 3.2 技术成熟度等级（TRL）

| 等级 | 描述 | 适用阶段 |
|------|------|----------|
| TRL 1-3 | 原理验证、概念实现 | 预研阶段 |
| TRL 4-5 | 实验室环境验证 | 可行性分析 |
| TRL 6-7 | 相关环境演示 | 产品开发 |
| TRL 8-9 | 系统完成并通过验证 | 量产阶段 |

### 3.3 路线选择决策流程
```
识别关键技术模块
    |
    v
列出可选技术路线（≥2条）
    |
    v
逐维度收集数据
    |
    v
评分与加权（建议≥3人独立评分）
    |
    v
识别关键差异项
    |
    v
风险评估与缓解方案
    |
    v
综合决策（非仅选分数高者，考虑战略匹配）
    |
    v
记录决策依据与备选方案
```

## 四、BOM估算方法

### 4.1 自上而下估算法
适用于早期阶段，快速估算总成本区间。

**方法**：
1. 确定目标售价与目标毛利率
2. 倒推可接受的总BOM成本 = 目标售价 × (1 - 毛利率)
3. 按模块/类别分配成本预算
4. 与自下而上估算交叉验证

**适用场景**：概念阶段、商业可行性快速判断
**精度**：±30%

### 4.2 自下而上估算法
适用于方案确定后，精确估算各器件成本。

**方法**：
1. 列出完整BOM清单（至最小可采购单元）
2. 逐条获取单价（公开报价/询价/历史数据）
3. 加总并增加损耗率（通常5-10%）
4. 加入NRE摊销（模具、认证、测试工装等）

**适用场景**：详细设计阶段、采购谈判准备
**精度**：±15%（询价后）/ ±5%（锁定价格后）

### 4.3 BOM成本分级估算表

```markdown
| 阶段 | 精度 | 方法 | 关键动作 | 责任人 |
|------|------|------|----------|--------|
| 概念 | ±30% | 自上而下+对标 | 参考同类产品BOM占比 | 系统工程师 |
| 方案 | ±20% | 自下而上+粗询价 | 核心器件公开报价 | 硬件工程师 |
| 详细设计 | ±15% | 完整BOM+正式询价 | 联系供应商获取报价 | 采购工程师 |
| 量产前 | ±5% | 锁定价格+协议 | 签订采购协议/备选方案 | 采购经理 |
```

### 4.4 成本敏感性分析

```markdown
| 器件类别 | 估算单价 | 数量 | 小计 | 占总成本比例 | 价格波动风险 |
|----------|----------|------|------|--------------|--------------|
| 主控芯片 | | 1 | | | 高/中/低 |
| 传感器 | | 2 | | | |
| 结构件 | | 1 | | | |
| 连接器 | | 5 | | | |
| 其他 | | | | | |
| **合计** | | | | 100% | |
```

**分析要点**：
- 识别占比>10%的高价值器件
- 对高价格风险器件制定备选方案
- 预留5-10%的成本缓冲应对价格波动

## 五、关键路径法（CPM）简介

### 5.1 基本概念
关键路径法（Critical Path Method）用于识别项目中最长的任务链，该链上的任何延迟都会导致项目整体延期。

### 5.2 实施步骤
1. **任务分解（WBS）**：列出所有技术验证与设计任务
2. **确定依赖关系**：明确任务间的前置/后置关系
3. **估算工期**：每个任务的最乐观/最可能/最悲观时间
4. **绘制网络图**：按依赖关系连接各任务节点
5. **计算路径**：找出总工期最长的路径（关键路径）
6. **识别浮动时间**：非关键路径上的可延迟时间

### 5.3 关键路径管理要点
- **聚焦资源**：将优质资源优先分配给关键路径任务
- **缓冲设置**：在关键路径末端设置项目缓冲（建议总工期的15-20%）
- **监控频率**：关键路径任务每日/每周跟踪，非关键路径可放宽
- **动态更新**：任务完成后重新计算关键路径，可能发生变化

### 5.4 技术项目关键路径常见节点
- 定制器件选型与样品获取
- 核心算法开发与仿真验证
- 结构模具设计与试模
- 长周期认证测试（安规/EMC/环境）
- 供应链首件确认与工艺验证
