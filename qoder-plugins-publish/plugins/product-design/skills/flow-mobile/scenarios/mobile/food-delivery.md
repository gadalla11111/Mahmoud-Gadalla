# Scenario: Mobile Food Delivery（移动端外卖/餐厅配送）

> **研究来源**：基于对 Drinkit（Food & Drink iOS App，最完整证据源）等真实 iOS 产品 flow 和页面的横向分析抽象，辅以 Julienne、SSENSE 通用电商 checkout 参考，不代表任何单一产品的具体实现。

---

## Identity

**Platform**: Mobile (H5 / React Native)
**Definition**: 以餐厅/商品浏览、定制化下单、在线支付为核心的移动端外卖/餐饮配送 App，用户通过地图或列表发现附近店铺，选购并定制商品，完成支付后实时追踪配送状态。

**Canonical Examples**: Uber Eats、DoorDash、Instacart、饿了么、美团

**Not this scenario if**:
- 产品是普通电商（卖实物商品，无「菜单式」商品定制，改用 mobile/marketplace）
- 产品是超市生鲜电商（无餐厅/店铺发现维度，改用 mobile/marketplace）
- 产品是线下餐厅内部点单系统（扫码自助，非配送场景）
- 产品是预约类餐厅（OpenTable 风格，改用 mobile/travel 的预订链路）

---

## User Profile

| 维度 | 内容 |
|---|---|
| **主要角色** | 订餐用户（选餐 + 下单 + 追踪配送）|
| **核心目标** | 快速找到想吃的餐厅 → 按喜好定制菜品 → 确认支付 → 知道外卖何时到达 |
| **心智模型** | 餐厅菜单的数字化版本：找店 → 看菜单 → 加购 → 结账，类比实体餐厅点餐流程 |
| **使用频率** | 中-高频（每周 2-5 次）；每次会话 5-15 分钟（浏览 + 下单）|
| **决策模式** | 即时欲望驱动：「现在想吃什么」为出发点，价格/口味/配送时间三要素决策 |
| **容错期望** | 低：下单前需明确确认商品+价格+地址；已支付订单修改困难，需 Dialog 确认不可逆操作 |

## IA Template

**导航模式**: Bottom Tab Bar（4 个 Tab）

```
Tab 1: 发现 / Discover — 地图 + 附近店铺列表（核心入口）
Tab 2: 订单 / Orders   — 历史订单 + 实时追踪
Tab 3: 收藏 / Saved    — 收藏的餐厅和菜品
Tab 4: 个人 / Profile  — 账户 + 地址 + 支付方式 + 设置
```

**选择建议**: 以配送为核心 → 地图/发现 Tab 作为主入口；以菜单浏览为核心（如单品牌 App）→ 菜单 Tab 为主入口，省略地图

**页面层级**: 3 级
```
L1: 发现（地图 + 店铺列表）
L2: 店铺菜单（商品列表）→ 商品详情（定制配置）
L3: 购物车 Bottom Sheet → 结账确认 Modal → 订单确认 → 实时追踪
```

**权限流结构**:
```
Location（位置，核心功能）:
  → App 启动时首次请求
  → 说明：「位置用于发现附近店铺和提供配送地址，不会在后台持续追踪」
  → 系统权限弹窗（「使用 App 期间允许」/ 「允许一次」/ 「不允许」）

Notifications（订单状态通知）:
  → 首次下单成功后触发（不在启动时强制询问）
  → 说明：「开启通知，及时收到「骑手已取餐」「外卖已送达」提醒」

Camera（商品扫描/二维码，可选）:
  → 部分 App 支持扫描菜单 QR，首次使用时请求
```

**数据密度**: 中-高
- 地图视图：店铺 Marker + 地图瓦片 + 底部店铺卡片 Bottom Sheet
- 店铺菜单：分类 Tab 横滑 + 商品卡片（图 + 名称 + 描述 + 价格 + 加购按钮）
- 商品定制：大图 + 营养信息 + 水平分类定制器 + 实时价格

**主要容器模式**:
| 场景 | 容器 |
|---|---|
| 地图点击店铺标记 | Bottom Sheet（Medium，店铺卡片）|
| 购物车管理 | Bottom Sheet（Large，随时可唤起）|
| 结账确认 | Modal（Large，在商品页上方，保留上下文）|
| 商品定制（轻量）| Bottom Sheet（Medium）|
| 商品详情（重量定制）| 全屏 Push 页 |
| 营业状态说明 | Inline Banner（红色/黄色，在购物车顶部）|
| 配送追踪 | 全屏地图 + 底部状态 Bottom Sheet |
| 联系骑手 | Action Sheet（电话 / 消息）|

## 该场景独有的 IA/UX 决策

1. **购物车必须以 Bottom Sheet 而非独立页面呈现，随时可唤起且不打断浏览** — 用户在浏览菜单时会多次加购，如果每次加购后跳转到独立购物车页，用户需要反复「返回」继续浏览，导致每次加购都有摩擦感和导航层级累积。正确做法：加购后底部出现悬浮购物车气泡（Floating Cart Bubble，含商品缩略图 + 总价），用户可随时点击气泡唤起购物车 Bottom Sheet 而无需跳转——整个购物过程始终在菜单视图上方叠加完成，保持「继续浏览」的流畅感（Drinkit flow 11003/11007 均采用此模式）。

2. **商品定制器必须支持实时价格反馈，每次选择后即时更新底部 CTA 上的价格** — 外卖商品的价格由基础价格 + 定制项加价共同构成（如「大杯 +¥5」「加浓 +¥3」），如果只在结账时显示最终价格，用户会对价格来源感到困惑，并对「多加了什么」产生不安全感。正确做法：商品详情页底部 CTA 上的价格随每次定制选择实时更新，让用户清晰看到「每次选择对价格的影响」——Drinkit（flow 11004，24 屏）是实时价格反馈的完整实现参考。

3. **营业状态必须在购物车和结账 CTA 三处前置显示，不能等用户点「结账」才报错** — 用户在店铺未营业时（凌晨、节假日）仍可能浏览菜单并加购，如果不提前告知营业状态，用户会在最后一刻（点「提交订单」时）才发现无法下单，造成强烈的挫败感。正确做法：① 店铺菜单顶部出现黄色/红色 Banner（「营业时间 09:00–21:00，现在未营业」）；② 购物车 Bottom Sheet 顶部显示红色状态 pill（「We open at 7:00 AM」）；③ 结账 CTA 按钮置灰并改为「暂未营业」——Drinkit（flow 11007）展示了购物车内的营业状态 pill 模式，CTA 明确显示「Not open yet」而非通用 disabled 样式。

4. **追加销售（Cross-sell）必须在加购后立即用半高 Bottom Sheet 展示，不能在结账时强推** — 推荐「配搭商品」的最佳时机是用户刚确认主商品的瞬间（此时用户决策成本最低），而不是在填写地址/支付信息时（此时用户的心理状态是「快点结束」，强推会引发反感）。正确做法：点击「加入购物车」后立即弹出半高 Bottom Sheet，展示 2-4 个推荐配搭（图片 + 价格 + 「+」按钮），用户可选择添加或直接关闭，关闭后不再出现——Drinkit（flow 11006）是此模式的标准实现，追加销售以「建议」而非「强制」形式出现。

5. **结账确认页必须明确显示 ETA（预计送达时间），而不只是「订单已接受」** — 用户在支付完成后最迫切想知道的不是「我订了什么」，而是「什么时候到」。模糊的确认文字（「您的订单已提交」）无法缓解用户的等待焦虑，会导致用户反复打开 App 查看。正确做法：订单确认页明确显示 ETA（「预计 25 分钟后送达」/ 「预计到达：14:35」），同时显示订单号和当前状态（「备餐中」），并引导用户进入实时追踪——Drinkit（flow 11012）在 Order Received 确认屏上明确显示 ETA 和状态阶段。

## Canonical Flows

> 以下 flow 基于 Drinkit 等真实产品样本横向分析抽象。括号内标注「行业共识」表示 3 个以上产品采用相同模式。

---

### Flow 1: 发现附近店铺（Store Discovery）

**在此场景的特殊性**: 外卖 App 的冷启动体验与其他 App 根本不同——第一次打开 App 时，位置权限和推送通知权限必须连续请求（没有位置就无法发现附近店铺）。地图是发现入口而非可选功能，店铺卡片（Bottom Sheet）是地图和菜单之间的关键过渡层。Drinkit（flow 10999）展示了「启动 → 权限 → 地图 → 店铺卡片 → 菜单」的完整 6 屏路径。

**行业共识**：外卖 App（Drinkit flow 10999）均在启动时请求位置权限（不可跳过），店铺详情通过点击地图 Marker 触发 Bottom Sheet（不直接跳菜单），Bottom Sheet 含「查看菜单」主 CTA。

**Entry**: App 启动 / Tab Bar「发现」Tab

```
Screen 1: 权限请求序列（首次启动）
  主操作: 授权位置以发现附近店铺
  关键组件（依次触发）:
    - 推送通知权限弹窗（系统弹窗，先于位置权限触发，可拒绝）:
        「[App名] 想给你发通知」→ 「允许」/ 「不允许」
    - 位置权限说明页（自定义页，系统弹窗前的 in-app 说明）:
        大图标（地图 Pin）
        标题「开启位置，发现附近店铺」
        说明「位置仅用于展示附近店铺和计算配送地址，不会在后台持续追踪」
        Button「允许使用位置」（主色）→ 触发系统位置权限弹窗
        Button「手动输入地址」（次要，灰色文字，跳过位置权限）
    - 系统位置权限弹窗: 「使用 App 期间允许」/ 「允许一次」/ 「不允许」
  → 授权成功: Screen 2（地图视图）
  → 拒绝位置: 显示地址输入框，用户手动输入地址后进入 Screen 2（无 Marker 地图或列表视图）

Screen 2: 地图发现视图（Map Discovery）
  主操作: 查找附近店铺
  关键组件:
    - 全屏地图（以用户位置为中心，蓝色圆点标记用户位置）
    - 店铺 Marker（地图 Pin，含店铺缩略图或分类图标）
    - 顶部搜索栏（「搜索菜品或店铺」，点击进入搜索 Screen）
    - 顶部地址栏（「当前位置：[地址]」，点击可修改配送地址）
    - 分类筛选 Chips（横向滚动，位于搜索栏下方）:
        「咖啡」/ 「快餐」/ 「日料」/ 「甜品」/ 「火锅」/ 「全部」
    - 列表/地图切换按钮（右上角）
    - [可选] 底部店铺列表（半高 Bottom Sheet，默认展示最近的 3-5 家店铺）
  → 点击店铺 Marker: 弹出店铺卡片（Screen 2a，Bottom Sheet）
  → 点击「列表」视图: Screen 2b（店铺列表视图）
  → 点击分类 Chip: 地图 Marker 过滤为该分类，列表同步过滤

Screen 2a: 店铺卡片（Store Card Bottom Sheet）
  主操作: 查看店铺信息，决定是否进入菜单
  容器: Bottom Sheet（Medium，地图仍可见于上方）
  关键组件:
    - 店铺封面图（宽图，Sheet 顶部）
    - 店铺名称（大号，左对齐）
    - 评分（⭐ 4.8 · 125 条评价）+ 分类标签（如「咖啡 · 轻食」）
    - 配送信息行: 📍 距离（「0.8km」）+ 🕐 预计配送时间（「20-30 分钟」）+ 💰 起送价（「¥0 起送」）
    - 营业状态（绿色「营业中」/ 红色「已关店，明天 09:00 开始」）
    - Button「查看菜单」（主色，全宽 CTA）
    - Button「收藏」（心形图标，右上角）
  → 点击「查看菜单」: Push → Screen 3（菜单页）
  → 点击外部地图区域: Sheet 收起，回到地图视图

Screen 2b: 店铺列表视图（可选，切换自地图视图）
  主操作: 浏览所有附近店铺
  关键组件:
    - 排序/筛选 Bar（评分 / 距离 / 配送时间）
    - StoreCard 列表（FlatList，下拉刷新）:
        封面图（横向矩形，16:9）
        店铺名 + 分类 + 评分 + 配送时间 + 起送价 + 距离
        Badge（「新店」/ 「热门」/ 「减配送费」）
  → 点击 StoreCard: Push → Screen 3（菜单页）

Screen 3: 店铺菜单页
  主操作: 浏览商品列表，点击添加到购物车
  关键组件:
    - NavigationBar: 返回 + 店铺名 + [❤ 收藏] [分享] [···]
    - 分类 Tab（横向滚动，固定在 NavigationBar 下方）:
        「全部」/「饮品」/「轻食」/「特选」/「季节限定」
    - 商品列表（按分类分组，SectionList）:
        每条商品行:
          商品图（右侧，80×80pt，圆角）
          商品名（左侧，加粗）
          描述（2行截断，灰色小字）
          价格（主色，左下角）
          Button「+」（右下角，圆形，点击直接加购最小单位）
    - 悬浮购物车气泡（有加购后出现，固定于底部 Tab Bar 上方）:
        左侧: 购物车图标 + 商品数量 Badge
        中间: 商品总价
        右侧: 「查看购物车」箭头
  → 点击商品行（非「+」区域）: Push → Screen 3a（商品详情/定制，Flow 2）
  → 点击「+」: 直接加购 1 份，购物车气泡出现/更新
  → 点击购物车气泡: 展开购物车 Bottom Sheet（Flow 3）
```

**Exit State**:
- ✅ 进入菜单：用户看到完整商品列表，购物车气泡初始为空/不显示
- ❌ 位置权限拒绝：显示手动地址输入，进入后仍可浏览店铺（按距离排序降级为按评分）
- ❌ 店铺已关店：查看菜单 CTA 变灰为「已关店」，进入菜单后显示红色 Banner，仍可加购（下单时提醒）

### Flow 2: 商品定制与加入购物车（Item Customization & Add-to-Cart）

**在此场景的特殊性**: 外卖商品的定制复杂度远超普通电商——温度（冰/热）、甜度（少糖/正常/多糖）、规格（大杯/中杯）、Add-on（加浓/加奶）是外卖 App 独有的定制维度。Drinkit（flow 11004，24 屏）是最完整的参考：水平可滚动分类卡片（每个分类展开后显示选项 + 步进控件）、互斥选项（Toggle 模式）、实时价格更新、营养数值同步更新。加购后立即推送追加销售 Bottom Sheet（flow 11006），是提升客单价的标准模式。

**行业共识**：Drinkit（flow 11004/11005）均采用「水平分类卡片 + 展开选项 + Stepper 步进」的定制器布局；加购后弹出 cross-sell Bottom Sheet（flow 11006）；底部 CTA 实时显示当前价格（非固定文字）。

**Entry**: 菜单页点击商品行（非「+」区域）

```
Screen 1: 商品详情页（Product Detail）
  主操作: 查看商品信息，开始定制
  关键组件:
    - 英雄图（全屏宽度，16:9 或正方形，可横向滑动查看多图）
    - 商品基础信息区:
        商品名称（大号，加粗）
        描述（完整，可展开）
        基础价格（主色，加粗）
    - 营养信息区（4 格横向排列，与定制项联动）:
        Energy（kcal）/ Protein（g）/ Fats（g）/ Carbs（g）
        颜色编码（能量红、蛋白绿、脂肪黄、碳水蓝）
        [条件更新] 每次定制选择后实时刷新数值
    - 定制器区（关键组件）:
        水平滚动分类 Tabs（「规格 Size」/ 「温度 Temp」/ 「甜度 Sugar」/ 「Add-ons」等）
        点击某分类 Tab → 下方显示该分类的选项卡片
        选项卡片（每个选项）:
          卡片样式（圆角矩形，选中: 白色背景 + 深色边框 + 勾选角标；未选: 灰色背景）
          选项名（如「大杯 16oz」）
          差价文字（如「+¥5」/ 「+¥0」）
          [互斥类型] RadioGroup（单选，如温度「冰 / 正常温 / 热」）
          [多选类型] 每个选项独立选中，Stepper 控制数量（- / 数字 / +）
    - 固定底部 CTA Bar:
        Button「加入购物车 ¥[实时价格]」（主色，全宽，价格随定制选择实时更新）
        [已选配置摘要，CTA 上方小字: 「大杯 · 少糖 · 热」]
  → 点击分类 Tab: 下方选项区切换，定制项更新
  → 完成选项选择: 底部价格更新，营养数值更新
  → 点击「加入购物车」: 商品加入购物车，进入 Screen 2（追加销售弹窗）

Screen 2: 追加销售（Cross-sell Bottom Sheet）
  主操作: 选择是否添加配搭商品
  容器: Bottom Sheet（Medium，商品详情页在后方仍可见）
  关键组件:
    - Sheet Header: 「常见搭配」/ 「你可能也喜欢」
    - 推荐商品网格（2×2 或横向滚动，3-4 个推荐）:
        每个商品 Card: 图片 + 名称 + 价格 + 「+」添加按钮
        [已选状态] 添加后按钮变为「✓」且填充颜色
    - Button「跳过」（右上角文字链接，灰色）或底部「完成」Button（添加选中商品）
    - [购物车气泡更新] 底部气泡同步显示商品数量和总价
  → 点击某商品「+」: 即时添加到购物车，商品 Card 变为已选状态

### Flow 3: 购物车管理（Cart Review & Edit）

**在此场景的特殊性**: 外卖购物车与电商购物车的最大差异是「营业状态」——用户可能在店铺未营业时完成了加购，需要在购物车层就提前告知「现在无法下单」，而不是等到结账时才报错。Drinkit（flow 11007）展示了购物车顶部红色营业状态 pill + CTA 置灰为「Not open yet」的正确模式。购物车中每个商品行需要显示完整的定制项（修改项清单），用户可直接在购物车内调整数量（Stepper）而无需返回商品详情。

**行业共识**：Drinkit（flow 11007）确认：购物车 Bottom Sheet 展示完整商品信息（含定制项）+ 营业状态前置 + 步进控件内联调整数量；左滑商品行显示删除按钮（swipe-to-delete）。

**Entry**: 点击购物车气泡 / 菜单页购物车图标

```

  主操作: 核查商品，调整数量，准备结账

    - Sheet Header（拖拽条 + 标题「购物车」+ Button「清空」文字链接，右侧）

### Flow 4: 快捷结账与订单追踪（Quick Checkout & Order Tracking）

**在此场景的特殊性**: 外卖结账的最终环节与电商不同——外卖不需要「选择收货地址」（已在启动时通过位置确定），而需要「确认配送地址是否准确」，并可选填「门牌号/楼层」等补充信息。结账完成后用户最关心的是 ETA（预计送达时间），Drinkit（flow 11012）在 Order Received 确认屏明确显示 ETA + 订单号 + 当前状态。订单追踪是外卖场景独有的实时地图体验——全屏地图 + 骑手位置实时更新 + 多阶段状态进度条 + Bottom Sheet 显示 ETA。

**行业共识**：Drinkit（flow 11012）确认：「Slide-to-Pay」或大 CTA 按钮结账；Order Received 确认页明确显示 ETA 和「Preparing」状态；支付完成后推送通知权限请求（如未授权）。

**Entry**: 购物车 Bottom Sheet → 「去结账」CTA

## Anti-Patterns

- **购物车是独立的 Tab 页面，加购后强制跳转**：用户每次加购都被带离菜单页，需要手动返回继续浏览，加购摩擦极高，导致用户「只选一样」不愿多浏览。→ 正确做法：购物车以 Bottom Sheet 呈现，加购后底部悬浮气泡更新数量和价格，用户点击气泡才展开购物车，浏览流程不被打断（Drinkit flow 11007）。

- **商品定制后底部 CTA 价格不实时更新**：用户选了「大杯（+¥5）」「加浓（+¥3）」后，CTA 按钮仍显示基础价格，结账时发现价格不同，产生「被欺骗」感。→ 正确做法：底部 CTA 价格随每次定制选择实时更新（「加入购物车 ¥36」），用户清楚知道每次选择对价格的影响（Drinkit flow 11004）。

- **店铺已关店时，用户点「结账」才提示错误**：用户花了 5 分钟挑选并加购，最后点击结账时才看到「商家暂不接单」，极其挫败。→ 正确做法：营业状态在购物车 Bottom Sheet 顶部前置显示（红色 Banner），CTA 按钮同步置灰并改为「暂未营业」文字，提前设置期望（Drinkit flow 11007）。

- **追加销售（Cross-sell）在结账确认页强推**：用户正在填写地址/支付时突然插入推荐商品，严重干扰结账流程，用户会选择忽略或感到厌烦，转化率极低。→ 正确做法：追加销售在加购后立即弹出半高 Bottom Sheet（非全屏，不完全阻断视图），用户可一键「跳过」，结账路径保持完整无干扰（Drinkit flow 11006）。

- **订单确认页只显示「订单已提交」，不显示 ETA**：用户支付完成后不知道外卖何时到，只能反复打开 App 查看，焦虑感极强。→ 正确做法：Order Received 确认页以最大字号显示 ETA（「预计 25 分钟后到达」），并引导用户进入实时追踪，让用户安心等待（Drinkit flow 11012）。

- **没有配送实时追踪，或追踪页无骑手位置**：用户不知道外卖在哪里、骑手是谁，焦虑感与上述等同。→ 正确做法：全屏地图展示骑手实时位置（每 15-30 秒更新）+ 多阶段状态进度条 + ETA 倒计时 + 联系骑手 CTA，让用户全程掌握配送进度。
