# Scenario: 内部运营工具（Internal Operations Tools）

## Identity
**Platform**: Web
**Definition**: 面向企业内部员工使用的运营管理系统，核心价值是将「审批流程」「工单追踪」「成员权限」通过状态机编排，让组织内的信息流动和操作授权有迹可循。
**Canonical Examples**: Aboard（员工请假 & 审批）、TravelPerk（差旅审批管理）、Mercury（支付审批）
**Not this scenario if**:
- 面向外部客户的工单系统（→ SaaS 管理后台）
- 纯数据监控看板（→ 数据分析 BI）
- 没有审批层级或状态机的简单任务列表（→ 通用 SaaS 管理后台）

---

## User Profile

| 维度 | 内容 |
|---|---|
| 主要角色 | Admin（系统管理员）/ Approver（审批人）/ Requester（申请人/普通操作员） |
| 核心目标 | **Admin**：配置审批规则、管理成员权限；**Approver**：快速处理待审批队列；**Requester**：提交申请并追踪进度 |
| 心智模型 | 「企业 ERP 的简化版」——用户期待严格的权限边界、完整的操作日志、可追溯的状态变更 |
| 使用频率 | Admin 低频配置 / Approver 高频日常处理 / Requester 按需提交 |
| 决策模式 | 任务驱动（Approver 需快速通过/拒绝）+ 配置驱动（Admin 按规则调整） |

## IA Template

**导航模式**: 左侧固定 Sidebar（220–260px）  
理由：内部运营工具功能模块多（审批、工单、成员、设置），需要持久可见的全局导航；用户是职能型使用者而非沉浸式用户，不需要全屏内容优先布局。

**页面层级**: Dashboard → List（审批队列/成员列表）→ Detail Panel（右侧叠加）→ Toast 反馈

**权限角色**:
- Admin：可见所有模块，含 Settings（规则配置 + 成员管理）
- Approver：可见审批队列 + 自己提交的申请，不可进入 Settings
- Requester：仅可见自己的申请列表 + 提交入口，审批动作不可见

**数据密度**: 高  
核心视图为多列 Table（申请人 / 类型 / 状态 / 审批链进度 / 时间），配合状态 Badge 快速扫描。

**主要容器模式**: 右侧 Detail Panel（Side Sheet）为主，Dialog Modal（邀请、确认类操作）为辅；避免全页跳转，保持列表上下文始终可见。

## 该场景独有的 IA/UX 决策

1. **Side Sheet 是审批队列的唯一合理 Detail 容器** — 审批者的核心工作模式是「连续处理多条待审记录」，类似邮件客户端的 Reading Pane 模式：左侧队列始终可见，右侧 Detail Panel 叠加展开（不跳转页面）。全页跳转打断「扫描→处理→下一条」的节奏——用户处理完一条后必须手动返回队列再定位到下一条，效率损耗极大。与 E-commerce / Consumer App 的「点击查看详情跳转页面」完全不同：那里每次决策是孤立的，内部审批工具的决策是流水线作业。

2. **审批链可视化 + 操作日志是合规基础设施，不是可选装饰** — 消费端 App 只需展示当前状态（「订单处理中」）；内部运营工具每一个审批动作都有法律合规和企业问责含义——谁批的、什么时间、批了还是拒了、附带什么备注，必须全程可查。Detail Panel 内的「历史日志区」（时间线：谁在何时做了什么操作）是业务要求，不是 nice-to-have。删掉操作日志等同于毁证——TravelPerk 的差旅审批流和 Mercury 的支付审批均强制保留完整审批时间线。

3. **批量审批确认必须呈现业务后果范围，不能只显示「N 条」** — 批量选择 + 底部浮动 Selection Bar 是审批队列的标准批量操作模式；但确认 Dialog 不能只显示「确认批准 5 条申请？」。报销类审批必须显示「合计金额（¥12,400）」，请假类审批必须显示「涉及总天数（15天）」——让审批者在点击「确认」前有实质性的决策依据，而非盲目批量通过。与电商批量删除商品（后果可逆）不同，批量审批的业务后果立即生效且通常不可逆。

4. **角色权限说明必须在授权界面同屏可见，不能只给外链** — 内部工具的角色授权一旦生效，Admin 可能访问财务数据、审批他人工单甚至修改系统配置——授权错误的纠错成本极高。Fingerprint（flow_id 11173）的双栏 Modal（左侧角色选择 + 右侧 Advanced Options 展开细粒度权限）是企业级最佳实践：即使是预设角色，也在右侧实时说明具体权限范围；有「Custom」选项的产品还允许细粒度按模块授权。对比 B2C 邀请流（只问「管理员还是普通成员？」），内部工具的授权粒度更细且后果更严重，不能简化成二选一。

5. **配置变更需配合「在途影响范围」提示，而非静默即时生效** — 审批规则变更与消费端「修改通知偏好」这类个人设置完全不同：规则一改，所有在途工单的路由逻辑立即变更——可能有 47 条已提交的申请会被路由到新审批人。变更保存时（不是保存后）必须提示「此变更将影响 X 条待审批申请」，让 Admin 了解操作的业务后果范围，而非点击「保存」后发现系统已静默重路由了所有在途请求。与 SaaS 设置变更（主要影响未来操作）不同，内部运营工具的配置变更往往对存量数据有即时影响。

## Canonical Flows

> 以下 flow 基于对真实产品的横向分析抽象而来，代表该场景的高频用户任务。

### Flow 1: 提交并处理审批请求（Submit & Process Approval Request）

**在此场景的特殊性**: 审批不是单一动作，而是有多级状态的过程。核心 IA 决策：审批 Detail Panel 必须在不离开列表主视图的前提下叠加展开（Side Sheet 模式），审批者需同时看到「上下文总览（谁、何时、是否有冲突）」和「操作区（Approve / Decline）」。与消费端 App 的全屏操作模式根本不同：操作粒度更细，支持批量选择 + 单项操作两种路径；状态机需在列表 Badge 上实时可见（Pending → Approved / Rejected）。

**前置条件**: 用户已登录；角色为 Approver（Requester 只能提交和查看自身申请，无 Approve 操作权限）；已有至少一条 Pending 状态的申请记录
**若前置条件不满足**: Requester 角色 → 「Approve / Decline」按钮不显示；待审批队列为空 → 空状态引导（不显示批量操作栏）；网络断连时审批操作提交失败 → Detail Panel 内联错误 + 重试按钮

**Entry**: Approver 在「审批管理 → 待审批」Tab 点击某条记录；或 Requester 点击「新建申请」

**Screens**:
```
Screen 1: 审批队列列表
  主操作: 点击某条记录（进入 Detail Panel）/ 「新建申请」
  关键组件:
    - Tabs（全部 / 待审批 / 已通过 / 已拒绝）
    - 多列 Table（☐ | 申请人 + 头像 | 申请类型 | 状态 Badge | 审批进度 | 提交时间）
    - 状态 Badge（●Pending 黄 / ●Approved 绿 / ●Rejected 红）
    - 审批链计数器（Approvals: 1 of 2）
    - 批量操作：Checkbox 多选 → 底部浮动 Selection Bar 出现
    - 搜索框 + 状态 Dropdown 筛选 + 时间范围筛选
  → 点击单条记录: 右侧 Detail Panel 滑入（Screen 2）
  → 批量勾选后点击 Approve: 确认 Dialog（是否批量通过 N 条）

Screen 2: 审批 Detail Panel（右侧滑出，Screen 1 列表保持可见）
  主操作: Approve / Decline
  关键组件:
    - 申请人信息（头像 + 姓名 + 部门 + 职级）
    - 申请详情区（类型、时间范围 / 金额 / 说明、附件 Chip）
    - 审批链可视化进度条（已批 N 级 / 共 M 级，每节点显示审批人姓名 + 时间）
    - Approve 按钮（绿色主按钮）+ Decline 按钮（Destructive 次按钮）
    - 可选备注输入框（拒绝时必填，审批时可选）
    - 历史日志区（时间线：谁在何时做了什么操作）
  → 点击 Approve: 按钮 Loading → Screen 3 成功状态
  → 点击 Decline: Panel 内联展开「拒绝原因」必填输入框 → 确认拒绝

Screen 3: 操作成功反馈（停留在 Screen 1 布局）
  主操作: 关闭 Panel，处理下一条
  关键组件:
    - Detail Panel 内 Badge 实时变为 Approved（绿）或 Rejected（红）
    - 底部左侧 Toast「申请已通过 / 已拒绝」（3 秒自动消失）
    - 列表对应行 Badge 同步更新
    - 可选：Panel 自动关闭，聚焦下一条待审批记录
  → 成功: Panel 留在已批记录或自动关闭，列表实时刷新
  → 错误（网络失败）: Screen 2 内 Panel 底部显示红色内联错误「提交失败，请重试」，按钮恢复可点击
```
**Exit State**: 申请状态在列表中变为对应 Badge；被审批方收到站内通知（状态变更）；审批历史日志记录本次操作和时间
**Empty State**: 「待审批」Tab 为空时显示「暂无待处理申请」+ 空状态插图，不显示 CTA（审批者无法主动创建申请）

### Flow 2: 配置审批规则与工作流（Configure Approval Workflow）

**在此场景的特殊性**: 这是「元层」配置操作——Admin 在配置的是「谁审批谁的什么请求、按什么条件触发」，而不是处理具体申请。核心 IA 决策：① 空状态必须提供引导式创建路径而非纯占位图（规则不存在 = 后续所有工单无法路由）；② 配置变更即时生效，需给出「生效范围」明确提示；③ 审批人选择器需支持按角色或具体人员两种维度，并实时预览权限说明。与 SaaS 通用 Settings 场景不同：此处的规则变更会即时影响其他用户的业务操作。

**前置条件**: 当前用户为 Admin 角色；已进入 Settings 页面（Approver/Requester 无法访问 Settings）
**若前置条件不满足**: Approver/Requester → Settings 入口不显示或 disabled + Tooltip 「需要 Admin 权限」；表单填写不完整 → 「创建规则」按钮保持 disabled；规则适用范围与现有规则冲突 → 内联 Warning Banner（非阻断）

**Entry**: Admin 进入 Settings → 审批管理 → 点击「+ 新建审批规则」

**Screens**:
```
Screen 1: 审批规则列表
  主操作: 「+ 新建审批规则」按钮
  关键组件:
    - 页面标题「审批管理」+ 说明文本（此规则控制工单路由）
    - Tabs（Active / Archived）+ Active 规则数量角标
    - 规则卡片（规则名称 + 适用范围摘要 + 审批人 + 创建时间 + 编辑/归档操作）
    - 空状态（无规则时）：居中引导卡片「设置审批流程控制您的工单路由」
      + 说明段落 + 「新建审批规则」居中主 CTA 按钮
  → 点击「新建审批规则」: Screen 2

Screen 2: 审批规则配置表单
  主操作: 「创建规则」保存按钮
  关键组件:
    - 规则名称输入框（必填）
    - 适用范围选择器（全员 / 指定部门 Multi-select / 指定人员 Multi-select）
    - 触发条件组（申请类型 Checkbox 组 + 可选金额/层级阈值输入）
    - 「添加审批人」区域:
        - 搜索框（按角色名 or 人名搜索）
        - 已选审批人列表（支持拖拽排序以设置审批顺序）
        - 右侧实时说明：当前悬停角色的权限范围文字
    - 审批模式 Radio（顺序审批 / 并行审批 / 任意一人即可）
    - 页脚：Cancel（次要文字按钮）+ 「创建规则」（填充主按钮，未填完时 Disabled）
  → 完整填写后 → 创建规则: Screen 3
  → Cancel: 返回 Screen 1（未保存更改提示 Dialog）

Screen 3: 规则创建成功（返回 Screen 1 列表）
  主操作: 查看新规则卡片
  关键组件:
    - 新规则卡片高亮出现在 Active 列表（短暂黄色边框动效）
    - Active 数量角标更新
    - 底部左侧 Toast「审批规则「[规则名]」已创建，立即生效」
  → 成功: 留在规则列表，可继续创建或编辑
  → 错误（规则冲突，适用范围重叠）: Screen 2 内联 Warning Banner「已存在适用此范围的规则，请调整适用范围」（非阻断，Admin 可强制保存）
```
**Exit State**: 新规则出现在 Active 列表中，系统立即按新规则路由后续工单审批请求；规则卡片显示「立即生效」状态提示
**Empty State**: 规则列表首次为空时显示引导卡片（带说明段落 + 居中 CTA），不使用简单「暂无数据」占位

### Flow 3: 邀请成员并分配操作权限（Invite Member & Assign Role）

**在此场景的特殊性**: 内部运营工具的成员邀请不只是「加人」，更是「授权」——被邀请人能访问哪些模块、能执行哪些操作，全由此步骤的角色配置决定。核心 IA 决策：① 角色说明必须在选择界面同屏可见（双栏 Modal：左侧选角色，右侧实时显示权限说明），避免 Admin 随意授权造成数据泄露；② 「Pending Invitations」需独立分区展示（区别于已接受的 Active 成员），支持撤回操作；③ 外部域名邀请需内联 Warning 提示（非阻断）。与 B2C 邀请流不同：粒度更细（按功能模块授权），且邀请后需等待对方接受才生效。

**前置条件**: 当前用户为 Admin 角色；已进入 Settings → 团队管理页面
**若前置条件不满足**: 非 Admin → 团队管理页面无「邀请成员」按钮；邮箱格式无效 → Modal 内 inline 错误；邮箱已存在于团队（Active 或 Pending）→ 内联提示「此邮箱已在团队中」，不重复发送

**Entry**: Admin 进入 Settings → 团队管理 → 点击「邀请成员」

**Screens**:
```
Screen 1: 团队成员列表（Settings 页面）
  主操作: 「邀请成员」主按钮（右上角）
  关键组件:
    - 成员 Table（头像 + 姓名 + 邮箱 + 角色 Badge + 最后活跃时间 + 状态）
    - 「Pending Invitations」独立分区（Badge 显示「Invited」橙色 + 发送时间）
      + 每条 Pending 项右侧「撤回邀请」次级操作
    - 搜索框（搜索已有成员）+ 角色 Dropdown 筛选
    - 每行 More 菜单（更改角色 / 停用账户）
  → 点击「邀请成员」: Screen 2 Modal 弹出

Screen 2: 邀请成员 Modal（双栏布局，居中弹出）
  主操作: 「发送邀请」主按钮
  关键组件:
    左栏（60%）:
      - Email 输入框（支持多邮箱，逗号分隔，Enter 确认为 Chip）
      - 「选择角色」标签 + 角色搜索框
      - 角色列表（分组 Checkbox）:
          管理员角色（全访问）/ 审批员（可处理审批）/ 操作员（只可提交申请）/ 只读（仅查看）
    右栏（40%，灰色背景卡片）:
      - 「[角色名] 权限说明」标题
      - 权限说明段落（当前 hover/选中角色的可见模块列表）
      - 「查看完整权限说明」链接
    底部（横跨双栏）:
      - 外部域名警告（如检测到非公司域名邮箱，显示内联 Warning，非阻断）
      - Cancel（次要）+ 「发送邀请」（填充主按钮）
  → 发送成功: Modal 关闭，Screen 3
  → 邮件格式错误: Modal 内 Email 输入框下方内联错误，不跳转

Screen 3: 成员列表更新（Screen 1 刷新）
  主操作: 无（自动反映结果）
  关键组件:
    - Pending Invitations 分区新增条目（邮箱 + 角色 + Invited Badge + 发送时间）
    - 底部 Toast「邀请已发送至 [email]」（有多条时显示「已发送 N 份邀请」）
  → 成功: 留在 Screen 1，可继续邀请
  → 错误（系统错误）: Toast 错误提示，Pending 列表不增加新条目；可重试
```
**Exit State**: 被邀请人邮箱出现在「Pending Invitations」分区，Badge 为「Invited」橙色；对方接受邀请后自动升级为 Active 成员，角色权限立即生效，Admin 收到系统通知
**Empty State**: Pending 分区为空时不显示分区标题（完全收起）；成员列表首次为空时显示「尚无团队成员」+ 「邀请第一个成员」CTA

### Flow 4: 批量审批处理与业务范围确认（Bulk Approval with Impact Summary）

**在此场景的特殊性**: 批量审批是审批者效率的核心路径，但与电商批量删除（后果可逆）不同，批量审批操作立即生效且通常不可逆——报销审批涉及真实资金划付，请假审批涉及排班影响。确认 Dialog 必须呈现业务后果汇总（合计金额/涉及天数），不能只显示「N 条」。Selection Bar 的浮动出现与隐藏时机（勾选时出现、全部取消后消失）也是核心交互细节。本 flow 基于 IA/UX 决策 #3 的要求，以及 Aboard（flow_id 771）展示的审批队列模式。

**行业共识**: 内部运营工具（Aboard、TravelPerk、Mercury）均采用 Table 多选 + 底部浮动 Selection Bar + Confirm Dialog 的批量操作模式。

**前置条件**: 用户已登录且角色为 Approver；「待审批」列表有 2 条以上 Pending 记录（单条走 Flow 1 个别审批路径）；记录类型为同质化可批量处理（同类型申请或混合类型但平台支持批量操作）
**若前置条件不满足**: 只有 1 条 Pending 记录 → 无法触发 Selection Bar 批量路径，仅走 Flow 1；Requester 角色 → Checkbox 不显示，无批量操作权限；已全部批准/拒绝 → 待审批 Tab 为空，显示空状态

**Entry**: Approver 在「审批管理 → 待审批」Tab 勾选多条记录

```text
Screen 1: 审批队列（多选模式）
  主操作: 勾选多条记录，触发底部浮动 Selection Bar
  关键组件:
    - 多列 Table（每行首列有 Checkbox）
    - 表头「☐ 全选」控制：全选当前页 / 取消全选
    - 底部浮动 Selection Bar（勾选 ≥ 1 条时滑入）:
        「N 项已选」计数 + [Approve N] 主按钮（绿色）+ [Decline N] 次按钮 + [×] 取消选择
    - 已选行高亮背景（蓝色浅底）
    - Selection Bar 不覆盖 Table 最后一行（底部留白适配）
  → 勾选后点击 [Approve N]: Screen 2（确认 Dialog）
  → 勾选后点击 [Decline N]: Screen 2b（拒绝原因 Dialog）
  → 点击 [×]: 取消所有选中，Selection Bar 消失

Screen 2: 批量批准确认 Dialog
  主操作: 确认批量批准，查看业务后果汇总后点击「Approve」
  关键组件:
    - Dialog 标题:「批准以下 N 条申请？」
    - 业务后果汇总（关键区分点）:
        请假类：「合计涉及 X 天请假（Jun 10 – Jun 21）」
        报销类：「合计批准金额：¥ X,XXX」
        混合类：按类型分行汇总
    - 申请明细列表（可折叠，每行：姓名 + 类型 + 关键数值）
    - [Cancel]（次级）+ [Approve N items]（绿色主 CTA）
    - 说明文案：「此操作批准后立即生效，无法撤销」
  → 点击「Approve N items」: 批量提交 → Loading → Screen 3
  → 点击「Cancel」: 关闭 Dialog，返回 Screen 1（选中状态保留）

Screen 2b: 批量拒绝原因 Dialog（可选分支）
  主操作: 填写统一拒绝原因，确认拒绝
  关键组件:
    - Dialog 标题:「拒绝 N 条申请」
    - 拒绝原因文本框（必填，此原因将统一发送给所有申请人）
    - 说明文案：「以下 N 位申请人将收到拒绝通知」+ 姓名列表
    - [Cancel] + [Decline N items]（红色 Destructive 主 CTA）
  → 填写原因后点击「Decline N items」: 批量提交 → Screen 3

Screen 3: 批量操作成功反馈
  主操作: 确认结果，继续处理剩余记录
  关键组件:
    - 底部 Toast「N 条申请已批准 / 已拒绝」（3-5 秒，含 [×]）
    - 列表实时刷新：已处理记录 Badge 更新（Approved/Rejected），或从「待审批」Tab 移除
    - Selection Bar 消失（选中状态清除）
    - 「待审批」Tab 角标数量更新（减去已处理数量）
  → 成功: 继续在列表中处理剩余 Pending 记录
  → 部分失败（如某条记录已被他人处理）: Toast 带 Warning「N 条已批准，X 条已由他人处理」+ 列表刷新
```

**Exit State**:
- ✅ 批量批准成功：N 条记录 Badge 变为 Approved（绿），申请人收到系统通知，审批日志记录批量操作者和时间
- ✅ 批量拒绝成功：N 条记录 Badge 变为 Rejected（红），申请人收到含拒绝原因的通知
- ⚠ 部分失败：Toast 显示「成功 N 条，失败 X 条」，列表重新加载最新状态

## Anti-Patterns

> 该场景中真实存在的常见设计错误，基于研究观察。

- **全页跳转处理审批**：审批者点击「查看」跳转到独立页面，离开队列视图 → 正确做法：Side Sheet / Detail Panel，保持列表上下文，审批完立即可处理下一条
- **空状态只显示「暂无数据」**：审批规则列表或成员列表为空时用通用占位图，Admin 不知道该做什么 → 正确做法：空状态必须包含引导文字 + 明确 CTA，说明「不配置此处会影响什么」
- **角色选择界面隐藏权限说明**：邀请时只展示角色名，点击才展开说明 → 正确做法：双栏 Modal，右侧实时显示当前选中角色的权限范围，帮助 Admin 做出正确授权决策
- **状态机只有两个状态**：工单/审批只有「进行中 / 完成」→ 正确做法：呈现完整状态节点（草稿→提交→待审→已审→已执行/已拒绝），并在审批链进度中可视化每个节点的负责人和时间
- **忽略 Pending Invitations 生命周期**：邀请发送后无法追踪、无法撤回 → 正确做法：独立展示 Pending 分区，提供撤回操作和发送时间记录，超时邀请需提示重发
- **批量审批确认仅显示条数不显示业务范围**：批量通过申请的确认弹窗只写「批准 5 条申请」，审批者无法判断实际影响 → 正确做法：确认 Dialog 必须呈现业务后果汇总（如「合计报销金额 ¥12,400」或「涉及总请假天数 15 天」），让审批者在点击确认前有实质性判断依据
- **角色提升静默生效无二次确认**：将普通成员提升为 Admin 后台即时生效，无任何确认步骤 → 正确做法：Admin 级别的角色变更需要 Destructive-level 确认 Dialog，明确列出「授予 [姓名] Admin 权限将允许访问：财务数据、审批管理、成员设置」，让 Admin 对权限变更的具体后果有清晰认知再操作
