# Scenario: Healthcare（医疗健康平台）

> **研究来源**：基于对 Acuity Scheduling、Fresha、Hers、time2book 等产品的 flow 横向分析抽象，不代表任何单一产品的具体实现。

---

## Identity

**Platform**: Web
**Definition**: 面向患者/用户的预约就诊平台，或面向医疗机构运营端的患者管理系统，核心围绕「预约管理 + 健康档案 + 医患沟通」三轴展开，并内嵌隐私合规约束（HIPAA/GDPR）。

**Canonical Examples**: 医鹿（健康预约平台）、Health on Demand（远程诊疗）、MindBody（健康服务预约）

**Not this scenario if**: 产品是健身追踪类移动端 App（改用 ios/health-fitness）；产品纯粹做医院挂号 HIS 系统（改用 web/internal-ops）；产品以医疗数据报告/BI 为主（改用 web/data-analytics）。

---

## User Profile

| 维度 | 内容 |
|---|---|
| **主要角色** | Patient / Client（患者，核心用户）/ Care Provider（医生/治疗师，查看患者端）/ Clinic Admin（诊所运营，管理日历）|
| **核心目标** | 患者：选择合适的诊疗服务 → 预约到合适时段 → 按时就诊；Admin：管理医生日历、查看患者档案 |
| **心智模型** | 兼具"消费者 App 易用性"期待（如网约车的预约体验）和"医疗机构专业感"期待（信任感、安全感、严肃感）|
| **使用频率** | 偶尔使用，每次就诊前触发一次完整流程 |
| **决策模式** | 任务驱动 + 高焦虑：用户在医疗场景中决策更谨慎，需要更多确认反馈和透明度 |
| **容错期望** | 极低容错：预约时间、服务类型、医生选择 的错误都可能造成实际就医损失 |

## IA Template

**导航模式**:
- **患者端（Patient-facing）**: 单列居中漏斗式布局，顶部线性进度条指示步骤，无侧边栏干扰，简洁引导用户完成任务
- **诊所管理端（Clinic Admin）**: Sidebar（左侧固定导航）+ 右侧内容主区，类 SaaS 管理后台布局

**页面层级**（患者端）:
```
L1: 健康摘要问卷（Intake Quiz）    ← 医疗场景专有，通用预约无此层
L2: 服务类型选择（Service Selection）
L3: 日期与时间选择（Date & Time Picker）
L4: 联系信息 + 隐私声明（Contact Info + Consent）
L5: 预约确认（Confirmation）
```

**权限角色结构**:
```
Patient       → 查看自己的预约、档案、消息；可发起预约、改期、取消
Care Provider → 查看被分配患者的档案和预约；可发送医患消息
Clinic Admin  → 全部操作（含日历管理、患者档案、财务报表）
```

**数据密度**:
- 患者端：低（每屏只做一件事，卡片式选项，无表格）
- 管理端：高（Table + Badge + 筛选，类 SaaS 后台）

**主要容器模式**:
| 场景 | 容器 |
|---|---|
| 取消 / 改期确认 | Dialog（AlertDialog，含取消政策） |
| 管理已有预约 | Modal 选项列表（Reschedule / Cancel） |
| 健康档案标签切换 | 同页 Tabs（无路由切换） |
| 多步预约漏斗 | 全屏单列线性 Wizard（无 Dialog，沉浸式） |
| 医患消息 | 右侧 Sheet 或独立页面（类 Email 三栏布局）|

## 该场景独有的 IA/UX 决策

1. **双界面 IA：患者端漏斗 vs 管理端 SaaS 后台，不可混用** — 医疗健康平台有两个截然不同的界面架构，必须分开设计：患者端（Patient-facing）用单列居中漏斗式 Wizard，每屏只做一件事，无 Sidebar、无 Table——这是「高焦虑、低频」场景，用户需要被引导；管理端（Clinic Admin）用 Sidebar + Table 的标准 SaaS 后台布局——这是「高频、任务驱动」场景，需要快速浏览和批量操作。将两种界面混为一谈（患者端用 SaaS 布局，或管理端用漏斗式）是严重的 IA 错误。Acuity Scheduling（flow_id 9866 患者端 vs flow_id 9854 管理端）是双界面参照的标准实现。

2. **健康问卷（Intake Quiz）必须先于服务选择，问诊结果决定服务可见性** — 医疗场景的服务选择依赖症状/需求上下文：用户不知道自己需要「初诊」还是「复诊」，不知道「内科咨询」和「全科体检」的适用场景。在展示服务列表前，必须先执行 1–3 题的健康摘要问卷（卡片式单选/多选），用问卷结果过滤或排序服务（如「根据您的症状，推荐以下服务」）。与通用预约平台（直接展示全量服务）的核心差异——Fresha 的「First-visit question」（flow_id 2379）是最轻量的问诊模式：一个问题即可将服务分级，适合首次预约场景。跳过问卷会导致患者选错服务类型，造成实际就医损失。

3. **时区是安全关键组件，必须贯穿预约全流程每个步骤** — 远程诊疗的时区不是用户「偏好设置」，而是安全关键：时区误解直接导致患者爽约，产生 No-show 费用，甚至延误诊疗。时区必须在以下位置均显示：① 日期时间选择屏（大号显著，而非小字脚注）；② 服务卡片的时段标注；③ 确认页预约摘要（如「2026年4月17日 10:00 AM，北京时间 GMT+8」）。必须提供修改入口（Timezone Selector，支持城市名搜索）。Acuity Scheduling（flow_id 9866）是贯穿全程时区显示的标准参照；Cal.com（flow_id 3497）是面向访客不强制登录的时区显示范本。

4. **取消流程必须展示取消政策，Destructive 确认前不可省略** — 医疗预约取消与通用预约（直接确认）有根本不同：医疗取消有法律和财务后果（No-show 费用、提前 N 小时取消规则），患者必须在确认前被告知。AlertDialog 中必须包含：① 高亮的取消政策摘要块（如「24 小时内取消将收取 ¥100 违约金」）；② Destructive 样式确认按钮（红色/黑色，表示不可逆）；③「改期」作为替代出口（与「取消」并列，让患者意识到还有其他选择）。Fresha（flow_id 2383）是正确实现参考：取消政策在确认前强制展示，避免事后纠纷。

5. **PHI 隐私同意必须内嵌表单步骤，不可用独立弹窗** — 医疗平台收集患者健康信息（PHI/PII）时，隐私告知文字和同意 Checkbox 必须内嵌在联系信息表单步骤内（紧跟手机号/Email 字段之后），不能用独立弹出的「隐私政策」弹窗或单独页面。原因：① 单独弹窗会被用户习惯性关闭，导致未真正知情同意；② HIPAA/GDPR 要求同意行为与数据收集行为在同一步骤发生；③ 内嵌 Checkbox（「我同意以上隐私政策，并授权将我的健康信息传输给诊疗提供方」）必须纳入表单验证（未勾选时提交按钮 disabled），确保合规——这是医疗场景独有的表单约束，通用预约平台（如 Cal.com）不要求此步骤。

## Canonical Flows

> 以下 flow 基于对真实产品的横向分析抽象而来，代表该场景的高频用户任务。

### Flow 1: 选择服务并预约时段（Book Appointment）

**在此场景的特殊性**: 相比通用预约平台直接展示服务列表，医疗场景必须先执行健康摘要问卷（Intake Quiz），用问卷结果过滤可用服务。时区选择器是远程诊疗的必须组件（时区误解 = 医疗安全问题）。联系信息步骤须嵌入隐私告知文字（PHI 数据使用说明 + 同意勾选项）。

**前置条件**: 无（任何访客均可开始预约流程，无需提前登录）
**若前置条件不满足**: 若要求登录才能预约，则在提交联系信息前触发注册/登录步骤；若医生日历为空（无可用时段），在 Step 3 显示空状态并引导选择其他医生或时段

**Entry**: 落地页"开始问诊"按钮 / 首页"预约医生"CTA

**Screens**:

```
Screen 1: 健康摘要问卷（Health Intake Quiz）
  主操作: 选择选项（单选/多选卡片）→ 自动跳至下一题
  关键组件: Progress Bar（步骤 X/总）、问题标题（Bold H2）、
            说明文字（Gray 辅助）、选项卡片列表（RadioGroup / CheckboxGroup）、
            返回箭头按钮
  → 完成所有问题: Screen 2
  → 提前退出: 返回首页

Screen 2: 服务类型选择（Service Selection）
  主操作: 点击服务卡片上的 BOOK 按钮
  关键组件: 服务卡片列表（Card: 服务名 + 时长 + 价格 + 标签 + BOOK Button）、
            分组标题（推荐服务 / 全部服务）、Timezone 当前显示（灰色辅助文字）
  → 选择服务: Screen 3
  → 取消: 返回首页

Screen 3: 日期与时间选择（Date & Time Picker）
  主操作: 点击月历日期 → 选择右侧时间槽 → 点击 NEXT
  关键组件: 已选服务摘要卡片（顶部）、Calendar 月视图（当前日期圆形高亮 / 不可用灰显）、
            可用时间槽网格（3列，30分钟间隔）、
            Timezone Selector（下拉：显示 GMT+XX:XX，支持修改）、
            NEXT 按钮（时间槽未选时 disabled）
  → 选择时间槽: Screen 4
  → 返回: Screen 2

Screen 4: 填写联系信息（Contact Information & Consent）
  主操作: 填写必填项 → 勾选隐私同意 → 点击 CONFIRM APPOINTMENT
  关键组件: 表单字段（First Name* / Last Name* / 手机号（含国际区号选择器）* / Email*）、
            隐私告知文字块（"您的医疗信息将安全传输给诊疗提供方，用于…"）、
            Checkbox（"我同意以上隐私政策"，必选）、
            折叠区：优惠码 / 保险套餐码（Coupon / Package Code）、
            Separator、提交按钮（Confirm Appointment）
  → 提交成功: Screen 5
  → 验证失败: Screen 4（字段内联错误提示）

Screen 5: 预约确认（Appointment Confirmed）
  主操作: 添加至日历 / 返回首页
  关键组件: 成功图标（绿色 CheckCircle）、预约摘要卡片（医生 / 服务 / 日期时间 / 时区）、
            Add to Google Calendar Button、Add to iCal/Outlook Button、
            文字链接（RESCHEDULE / CANCEL），触发 Flow 3、
            "再预约一次" 导航链接
  → 添加至日历: 系统日历
  → 取消/改期: Flow 3
  → 再预约: Screen 1
```

**Exit State**: 用户看到绿色确认屏，预约详情（日期、时间、医生、时区）清晰呈现；系统同步发送邮件确认
**Empty State**: 若所选日期无可用时段，时间槽区域显示"该日期暂无可用时段，请选择其他日期"（不隐藏日历）

### Flow 2: 查看健康档案与医患消息（View Health Records & Messaging）

**在此场景的特殊性**: 医疗档案与通用 CRM 客户档案的关键差异在于：需要呈现知情同意书签署状态、病史表单完成状态；「与医生发消息」是档案页核心入口而非辅助功能；处方记录、实验室报告需要分区展示并支持 PDF 下载。HIPAA 要求对档案访问做日志留存（技术层面，UI 无需展示）。

**前置条件**: 用户已登录（健康档案为隐私数据，不可匿名访问）；账户下已有预约历史或填写过健康问卷
**若前置条件不满足**: 未登录用户访问档案页 → 重定向至登录页；新账户无任何记录 → 各 Tab 显示空状态 + 预约引导 CTA

**Entry**: 登录后用户 Dashboard / 顶部导航"健康档案"

**Screens**:

```
Screen 1: 患者仪表盘（Patient Dashboard）
  主操作: 查看待办提醒 / 进入各功能区
  关键组件: 顶部导航（Action Items 角标 / Messages / Appointments / Profile）、
            橙色 Banner（待办提醒："完善您的健康问卷"，带 CTA）、
            快速入口卡片（即将到来的预约 / 未读消息数 / 处方续药提醒）
  → 点击 Messages: Screen 2
  → 点击 Appointments: Flow 1 或 Flow 3
  → 点击 Health Records: Screen 4

Screen 2: 消息中心（Messages）
  主操作: 点击消息查看对话
  关键组件: 消息列表（Avatar + 类别标签（如"Birth Control"）+ 消息预览 + 时间戳）、
            未读消息蓝色圆点、分类筛选 Tabs（全部 / 医疗团队 / 系统通知）、
            Empty State（"暂无消息，预约后可与医疗团队沟通"）
  → 点击消息: Screen 3
  → 返回: Screen 1

Screen 3: 医患对话（Conversation）
  主操作: 输入消息 → 发送
  关键组件: 对话气泡（患者消息右对齐、蓝色；医生消息左对齐、灰色）、
            医生 Avatar + 认证标识（Verified Badge）、
            消息输入框 + 附件上传按钮（上传检验报告图片）、
            快捷回复建议
  → 发送成功: 新气泡出现
  → 返回: Screen 2

Screen 4: 健康档案（Health Records）
  主操作: 在各标签页查看记录 / 下载 PDF
  关键组件: Tabs（就诊历史 / 处方记录 / 实验室报告 / 知情同意书）、
            记录列表（日期 + 类型 + 状态 Badge：Completed 绿 / Pending 橙）、
            每条记录右侧 Download PDF Button、
            知情同意书项目（Completed 打勾 / Pending 警告图标 + "签署"CTA）
  → 下载: 文件下载（浏览器原生）
  → 返回: Screen 1
```

**Exit State**: 用户完整查看档案并与医疗团队建立消息沟通，或成功下载所需健康报告
**Empty State**: 各标签页初次访问时显示"暂无[类型]记录"，提供"预约医生"入口（而非空白占位）

### Flow 3: 取消或改期已有预约（Cancel or Reschedule Appointment）

**在此场景的特殊性**: 与通用预约取消（直接确认）不同，医疗取消流程有三个强制要素：1) 取消前必须展示取消政策（Cancellation Policy），说明 No-show 费用是否适用；2) 取消后必须提供"立即改期"的主 CTA（避免患者因取消而延误就医）；3) 在终态页面提供紧急就医导引（症状加重的急救指引），体现医疗场景的安全责任。

**前置条件**: 用户已登录；预约状态为 Confirmed（已取消或已完成的预约不可再次操作）；预约时间尚未到来
**若前置条件不满足**: 已过时间的预约 → Manage Appointment 按钮不显示（或禁用）；已取消的预约 → 详情页仅显示 BOOK AGAIN CTA

**Entry**: 导航"我的预约" → 选择一条状态为 Confirmed 的预约

**Screens**:

```
Screen 1: 预约列表（Appointments List）
  主操作: 点击目标预约查看详情
  关键组件: Tabs（Upcoming / Past / Cancelled）+ 数量徽章、
            预约卡片列表（服务名 + 日期时间 + 医生 + 价格 + 状态 Badge）、
            搜索 / 筛选栏、
            Empty State（"暂无预约"+ "去预约"CTA）
  → 选择预约: Screen 2
  → 列表空: Empty State

Screen 2: 预约详情面板（Appointment Detail）
  主操作: 点击 MANAGE APPOINTMENT
  关键组件: 状态 Badge（Confirmed 紫 / Cancelled 红色）、
            日期时间（大号加粗）、服务名称 + 医生名 + 机构名、
            操作行列表（Add to Calendar / Manage Appointment）、
            服务费用明细（项目 + 价格 + 折扣 + 总计）、
            知情同意书状态（已完成 ✓ / 待签署 ⚠）、
            取消政策说明文字段（灰色小字）
  → 点击 Manage Appointment: Screen 3
  → 点击 Add to Calendar: 系统日历

Screen 3: 管理预约 Modal（Manage Appointment）
  主操作: 选择 RESCHEDULE 或 CANCEL
  关键组件: Modal 顶部预约摘要（服务名 + 日期时间）、
            RESCHEDULE APPOINTMENT 选项行（CalendarIcon + 文字）、
            CANCEL APPOINTMENT 选项行（XIcon + 文字，色调较弱）、
            关闭按钮（X）
  → 点击 Reschedule: 跳转 Flow 1 Screen 3（带已选服务预填）
  → 点击 Cancel: Screen 4

Screen 4: 取消确认对话框（Cancel Confirmation）
  主操作: 点击"确认取消"
  关键组件: AlertDialog 标题"确认取消此预约？"、
            说明文字（"如需更换时间可选择改期"）、
            取消政策告知（高亮块：如"12小时内取消将收取 ¥XX 违约金"）、
            确认取消 Button（Destructive 样式：红色/黑色）、
            返回按钮（关闭 Modal）
  → 确认取消: Screen 5
  → 返回: Screen 3

Screen 5: 取消成功状态（Cancelled）
  主操作: 点击 BOOK AGAIN 重新预约
  关键组件: Cancelled 红色 Badge、划线日期时间（表示已失效）、
            BOOK AGAIN 主 CTA 按钮（靠近顶部）、
            预约历史自动归档至"Past"Tabs、
            紧急就医提示（"如症状紧急，请拨打急救电话 120 或前往最近急诊"）
  → 点击 Book Again: Flow 1 Screen 1
  → 退出: 返回预约列表
```

**Exit State**: 预约状态更新为红色 Cancelled Badge，条目移至 Past 分类，页面提供"BOOK AGAIN"CTA 和紧急就医引导
**Empty State**: 取消确认 Dialog 内若取消政策为空，不显示政策区块（不留空占位）

### Flow 4: 取消预约 + 退款流程

**在此场景的特殊性**: 此 flow 是 Flow 3 的深化版本，专注于「已付款预约的取消与退款」这一医疗场景独有问题。通用预约取消（Flow 3）聚焦于取消本身；本 flow 加入退款政策审阅、退款金额确认、退款状态追踪三个医疗付费场景必须的节点。Fresha（flow_id 2383）是含取消政策的完整参考实现，Acuity Scheduling（flow_id 9866）展示了确认后的退款状态处理模式。退款处理时间（1-5 个工作日）与「退款已发起」Toast 的分离呈现，是降低患者焦虑的关键设计。

**前置条件**: 用户已登录；预约为已付款状态（支付了预约费用）；预约时间在取消政策允许的退款窗口内（如：预约时间 24 小时前取消可退款）
**若前置条件不满足**: 在退款窗口外取消 → AlertDialog 提示「超出退款窗口期，取消不可退款」，但仍可取消（无退款）；预约为免费或已完成 → 无退款流程，走 Flow 3 普通取消

**Entry**: 预约列表 → 点击已付款的 Confirmed 预约 → Manage Appointment → Cancel

**Screens**:

```
Screen 1: 预约详情（已付款，Confirmed 状态）
  视觉状态: 预约卡片显示付款金额和付款方式；取消政策在详情下方灰色小字展示
  关键组件:
    - 预约摘要: 服务名 + 医生 + 日期时间 + 时区
    - 付款记录: 「已支付 ¥280.00 · Visa 末尾 4242 · Apr 10」
    - 取消政策说明:「24 小时前取消：全额退款；12-24 小时：退 50%；12 小时内：不退款」
    - [MANAGE APPOINTMENT]  主 CTA
    - [Add to Calendar]  次要 CTA
  → 点击 MANAGE APPOINTMENT: Screen 2

Screen 2: 管理预约 Modal（含退款预估）
  视觉状态: Modal 显示两个操作选项；改期选项置于首位；取消选项下方有退款预估金额
  关键组件:
    - [📅 RESCHEDULE APPOINTMENT]（改期，首选引导）
    - [✕ CANCEL APPOINTMENT]（取消）
      └ 次级文字:「根据取消政策，您将获得退款 ¥280.00」（在退款窗口内时显示）
      └ 若超出退款窗口:「超出退款窗口期，取消不退款」（红色警告文字）
    - [×] 关闭
  → 点击 CANCEL: Screen 3
  → 点击 RESCHEDULE: 跳转 Flow 1 Screen 3

Screen 3: 取消与退款确认 AlertDialog
  视觉状态: AlertDialog 居中，高亮政策块 + 退款金额确认
  关键组件:
    - 标题:「确认取消此预约？」
    - ⚠ 取消政策摘要块（背景色高亮）:
        「根据取消政策，在预约 24 小时前取消：全额退款 ¥280.00」
        「退款将在 5-7 个工作日内退回原付款方式（Visa 末尾 4242）」
    - 退款明细:「预约费 ¥280.00 → 退款 ¥280.00 · 手续费 ¥0」
    - [返回改期]（次要，减少直接取消）
    - [确认取消并申请退款]（Destructive 样式，红色）
  → 点击确认: Screen 4

Screen 4: 取消成功 + 退款发起状态
  视觉状态: 预约 Badge 变为 Cancelled；底部 Toast 提示退款处理中；预约移至 Past 分类
  关键组件:
    - 状态 Badge:「Cancelled」（红色）
    - 划线日期时间（表示已失效）
    - Toast（底部，5 秒）:「✓ 预约已取消，退款 ¥280.00 正在处理中」
    - 退款状态追踪区块:
        「退款申请号: #REF-4729」
        「预计到账: 5-7 个工作日（Visa 末尾 4242）」
        [查看退款状态]（链接，跳退款追踪页或支持页）
    - [BOOK AGAIN]  主 CTA（重新预约引导）
    - 紧急就医提示:「如症状紧急，请拨打急救电话 120 或前往最近急诊」
```

**Exit State**:

- 退款窗口内取消 → 预约取消，退款申请已发起，显示退款申请号和预计到账时间
- 退款窗口外取消 → 预约取消，无退款，显示「无退款（超出取消政策窗口期）」说明
- 放弃取消 → 点击「返回改期」或关闭 AlertDialog，预约状态不变

**Empty State**: N/A（此 flow 针对特定的已付款预约，无空状态）

## Anti-Patterns

> 该场景中真实存在的常见设计错误，基于研究观察。

- **跳过健康问卷直接展示服务列表**：医疗场景的服务选择依赖症状/需求上下文，跳过问卷会导致患者选错服务类型 → 在服务入口前嵌入 Intake Quiz，问卷结果过滤服务列表

- **时区未显示或不可修改**：远程诊疗跨时区时，时区误解直接导致患者爽约 → 在日期时间选择屏显著展示当前时区，并提供修改入口

- **取消流程无取消政策说明**：患者不清楚 No-show 规则，取消后产生纠纷 → 取消确认 Dialog 中强制展示取消政策摘要（HIPAA 合规也要求书面告知）

- **取消终态无改期 CTA**：患者取消后无明确引导，可能延误就医 → 取消成功页必须提供"BOOK AGAIN"为首要 CTA，紧急情况提供急救联系方式

- **隐私告知与表单分离（单独弹出）**：用户忽略单独弹出的隐私政策，合规风险高 → 将隐私告知文字块内嵌于联系信息表单步骤中，Checkbox 同意项紧跟其后

- **健康档案仅展示列表无状态标识**：患者不清楚哪些知情同意书待签署 → 每条档案记录配合状态 Badge（Completed 绿 / Pending 橙），待签署项目提供内联 CTA
